# Requirements — Phone Timelapse

## Overview

A local-network tool that uses an iPhone (Safari) as a remote camera for capturing long-duration timelapses (up to 12 hours). The phone streams its rear camera feed; a Node.js server on the PC sends timed capture commands and saves frames to disk. A browser dashboard on the PC controls the session and displays live status.

No external services. No cloud. Phone and PC must be on the same WiFi network.

## Functional Requirements

### FR-1 — Capture

| ID | Requirement |
|----|-------------|
| FR-1.1 | The server sends a "capture now" command to the phone over WebSocket at a configurable interval (default: 30 seconds). |
| FR-1.2 | The phone captures a still frame from its **rear camera** via a canvas snapshot of a live video element. |
| FR-1.3 | The captured frame is transmitted to the server over WebSocket. |
| FR-1.4 | The server saves the frame to disk as PNG or JPG (configurable per session). |
| FR-1.5 | No images are saved to the phone's camera roll or local storage. |
| FR-1.6 | The phone page forces **landscape orientation** and uses the rear-facing camera (`facingMode: 'environment'`). |

### FR-2 — Session Management

| ID | Requirement |
|----|-------------|
| FR-2.1 | Sessions are started and stopped manually from the PC dashboard. |
| FR-2.2 | Sessions can also be started and stopped from the phone page. |
| FR-2.3 | A session can be scheduled to auto-start at a configured time of day. |
| FR-2.4 | A session auto-stops at a configured end time. |
| FR-2.5 | Default session window is 06:00–18:00 (configurable). |
| FR-2.6 | If a second session runs on the same calendar date, its output folder is named `YYYY-MM-DD (2)/`, then `(3)`, etc. |

### FR-3 — Output

| ID | Requirement |
|----|-------------|
| FR-3.1 | Frames are saved to `./output/YYYY-MM-DD/frame_0001.png` (or `.jpg`). |
| FR-3.2 | Frame numbering is sequential and monotonically increasing within a session. |
| FR-3.3 | If the phone disconnects and reconnects mid-session, frame numbering continues from the next number (no reset). |
| FR-3.4 | A session log (`session.log`) is written to the output folder when the session ends. |
| FR-3.5 | The session log includes: start time, end time, duration, frames captured vs. expected, and a list of all gaps with timestamps. |

### FR-4 — Phone Page

| ID | Requirement |
|----|-------------|
| FR-4.1 | The phone page is accessible via Safari at `http://[PC-LOCAL-IP]:3000/phone`. |
| FR-4.2 | Opening the page shows a single "Arm Camera" tap target (required for iOS to grant camera + wake-lock permissions). |
| FR-4.3 | After arming, the page shows the live rear camera feed as a background and a minimal status overlay. |
| FR-4.4 | Status overlay shows: countdown to next capture, timestamp of last successful capture, and connection status. |
| FR-4.5 | A subtle confirmation message ("Frame saved") fades out after each successful server acknowledgement. |
| FR-4.6 | The page uses a wake-lock technique (silent looping video) to prevent iOS from sleeping the screen. |
| FR-4.7 | If the WebSocket connection drops, the phone auto-reconnects and shows a visible "Reconnecting…" notice. |
| FR-4.8 | After reconnection, the phone resumes normal capture without user intervention. |
| FR-4.9 | The phone page has a Stop button that ends the session (same effect as stopping from the dashboard). |

### FR-5 — PC Dashboard

| ID | Requirement |
|----|-------------|
| FR-5.1 | The dashboard is accessible at `http://localhost:3000` (or `http://[PC-LOCAL-IP]:3000`). |
| FR-5.2 | The dashboard displays a QR code that opens the phone page URL directly. |
| FR-5.3 | The dashboard shows the latest captured frame as a thumbnail, updated after each capture. |
| FR-5.4 | The dashboard shows: frame count, countdown to next capture, session progress bar, estimated disk usage. |
| FR-5.5 | The dashboard has a gap/error log panel listing all disconnections and missed frames with timestamps. |
| FR-5.6 | The dashboard has a Settings panel with the following controls: capture interval, start time, end time, output format (PNG/JPG), and disk space estimate. |
| FR-5.7 | When the output format toggle is changed, the UI immediately shows an updated disk space estimate. |
| FR-5.8 | The dashboard has Start and Stop session buttons. |
| FR-5.9 | When a session ends, a summary is displayed in the dashboard. |

### FR-6 — HTTPS & Security

| ID | Requirement |
|----|-------------|
| FR-6.1 | The server runs over HTTPS using certificates generated by mkcert. |
| FR-6.2 | If mkcert certificate files (`cert.pem`, `key.pem`) are not found at startup, the server prints clear setup instructions and exits. |
| FR-6.3 | The dashboard displays a warning banner if accessed over plain HTTP (not HTTPS and not localhost). |

## Non-Functional Requirements

| ID | Requirement |
|----|-------------|
| NFR-1 | The server runs on Node.js in WSL (Windows Subsystem for Linux). |
| NFR-2 | No build step is required to run the project (`node server.js` starts everything). |
| NFR-3 | The phone and PC must be on the same WiFi network. This requirement is documented and displayed in the UI. |
| NFR-4 | The system must be stable for sessions up to 12 hours. |
| NFR-5 | No external/cloud services are used. All communication is on the local network. |
| NFR-6 | The tech stack is: Node.js, Express, Socket.IO, vanilla JS (no frontend build tool). |

## Decisions Made During Design

| Decision | Choice | Rationale |
|----------|--------|-----------|
| HTTPS strategy | mkcert with one-time iPhone cert install | Only reliable option for getUserMedia on iOS Safari over local network |
| WSL networking | Mirrored networking (already configured in .wslconfig) | Phone can reach WSL server directly via Windows WiFi IP |
| Socket.IO frame size | maxHttpBufferSize: 10 MB | Default 1 MB would silently drop PNG frames (3–5 MB each) |
| iOS orientation lock | CSS `@media (orientation: portrait)` overlay, no JS | screen.orientation.lock() unsupported on iOS Safari |
| JPG quality | 0.85 fixed | Visually excellent, ~25% smaller than default 0.92; PNG is lossless with no quality param |
| NoSleep MP4 | `public/nosleep.mp4` committed to repo, 2×2 px H.264 | 1×1 is invalid for H.264 (requires even dimensions) |
| Phone state before session | Camera preview visible, "Waiting for session to start…" overlay | Allows framing the shot before committing to a session |
| Scheduled start in past | Wait until tomorrow, show "Will start tomorrow at HH:MM" | Predictable; "start immediately" could surprise user on accidental past time |
| Concurrent frame handling | Skip new capture if previous toBlob in progress | Safest for timelapse; gaps are already logged; backlog would grow unbounded |
| Interval < 5s | Warning shown, not blocked | User may need short intervals for testing; warning is informative |
| iOS compatibility | FileReader for Blob→ArrayBuffer (works iOS 11+) | blob.arrayBuffer() added in iOS 14; FileReader works from iOS 11 |

## Known Constraints

### Camera Resolution via Browser

`getUserMedia` on iOS Safari does **not** give access to the full native camera sensor resolution (e.g. 12 MP). The browser video stream is capped at approximately **1920×1080 (1080p)**. This is a platform limitation, not a software choice. For most timelapse purposes, 1080p is excellent quality.

If full native sensor resolution is ever required, a native iOS app would be needed instead.

### Wake Lock on iOS

iOS Safari does not support the Web Screen Wake Lock API. The workaround (playing a tiny, muted, looping `<video>` element) is the best available browser-based approach but is not 100% guaranteed across all iOS versions. If the screen does lock, the page auto-reconnects when unlocked.

### Disk Space Estimates (reference)

At 30s interval over 12 hours = **1,440 frames**.

| Format | Approx. per frame | Total (12h, 30s) |
|--------|-------------------|------------------|
| PNG (1080p) | ~3–5 MB | ~4.3–7.2 GB |
| JPG (1080p) | ~0.5–1.5 MB | ~0.7–2.2 GB |

These are estimates shown in the settings UI to help the user choose format.
